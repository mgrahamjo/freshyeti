Controllers

The purpose of a controller is to get the data necessary to render a template. It does this by creating a context object and passing it to response.resolve(). When creating a context object, a controller has access to stored data as well as a number of data sources on the request, such as the session, query string, POST data, and the URL itself.

Controllers are passed to the configuration object on breadbox.init. For readability, you'll want to use require or import to store them as modules in their own files:

const breadbox = require('breadbox'); breadbox.init({ controllers: { '/index': require('./controllers/index'), '/docs': require('./controllers/docs') } });

Routes

The keys on the controllers object are used to match a requested URL to a particular controller function. Because all controllers match HTML views, '.html' extensions are ignored. For example, 'http://example.com/page.html' and 'http://example.com/page' would both match a controller with the key '/page'. In either case, by default breadbox would look for the corresponding view at /views/page.html. The url '/' is normalized to match the controller key '/index', and correspondes to the view at /views/index.html.

Authentication

If a user must be logged in to see a particular route, append the route key with '|authenticate'. Unauthenticated requests to the page will then automatically redirect to /login (assuming you haven't overwritten settings.loginPage), where a successful login attempt will redirect back to the correct route.

const breadbox = require('breadbox'); breadbox.init({ controllers: { '/change-password|authenticate': require('./controllers/change-password') } });

After authentication, user data can be found in request.sess.

Dynamic Routes

Often, parts of a URL are required but dynamic. For example, a typical blog post URL might look like 'http://example.com/posts/this-is-a-post-id'. In this case, we want to use the same controller no matter the post ID is, and we need to make the post ID available to the controller so that it can get the post from a data store. Breadbox's solution for this to use route parameters (not to be confused with query strings).

const breadbox = require('breadbox'); breadbox.init({ controllers: { '/posts/{{id|skip}}': require('./controllers/post') } });

Now, the post ID will be made available to the controller as request.params.id.

Response

The value of each property in the controllers object is a function. The first argument passed to this function is a response object. This object is a promise that can be resolved with three values:

  • context (object): Keys on this object will be available as variables in the view.
  • template (string): The path to the view template to use. This can be an absolute path, or else it will be treated as relative to /views/. The default template path will mirror the route.
  • headers (object): Any headers you want to send with the response. Sending a headers object does not overwrite the default headers object - any headers you send will be merged with default headers.
var breadbox = require('breadbox'); breadbox.init({ controllers: { '/index': function(response) { response.resolve( { context: "Foo" }, 'home.html', { 'Set-Cookie': 'bar=baz' } ); } } });

Request

The second argument passed to each controller function is a request object. This object has a number of properties that can give you information about the user and their request, as well as methods for doing cool stuff:

  • request.data (object): the raw request data from the browser.
  • request.body (object): POST or PUT data.
  • request.files (object): Any PUT or POSTed files.
  • request.params (object): Any dynamic portions of the requested URL. For example, if the controller key was '/posts/{{id|skip}}', the current post ID can be found on request.params.id.
  • request.cookies (object): Any cookies your app has set in the requesting browser.
  • request.redirect( [string] location, ([number] status), ([object] headers) ) (function): A function that can be used to send a redirect rather than resolving the response. Accepts three arguments:
    • location (string): A URL to be sent as the 'Location' header.
    • status (string): An HTTP status code. Defaults to 302
    • headers (object): Headers. Defaults to the default headers plus Location.
  • request.settings (object): The configuration object you passed to breadbox.init.
  • request.session (object): An object that has methods for session management. Breadbox ships with an in-memory session store, which makes it very fast, but less scalable than a database session solution. Unlike most in-memory session managers, Breadbox is able to persist sessions through a server restart or crash by dumping them to a JSON file before exiting.
    • session.save( [string] id, [any] data, ([string] key) ) (function): If key is provided, sets a key property with the value data on the session identified by id. If key is not provided, replaces the whole session object with data.
    • session.end( [string] id ) (function): Deletes the session.
    • session.get( [string] id ) (function): Returns the session.
    • session.all() (function): Returns an object containing all active sessions.
  • request.sess (object): Shorthand for request.session.get(request.cookies.id). The current user's session.
